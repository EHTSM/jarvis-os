# Phase 3 — External Connector Reality Verification

**Date:** 2026-07-17
**Mode:** Zero-optimism source verification + live server probes (no new integration code written).
**Server tested:** `PORT=5098 JWT_SECRET=verify-test-secret node backend/server.js`, authed via a self-signed HS256 JWT delivered as the `jarvis_auth` cookie (auth is cookie-based, not `Authorization:` header — a Bearer header returns 401).

## Method notes
- Every connector in `backend/services/integrationConnectors.cjs` uses a single real `_probe()` (raw `http`/`https` GET, `integrationConnectors.cjs:98`) against the vendor's real API host. No stub/mock probe path exists — connectors that lack credentials return `READY` and never fake a `CONNECTED`.
- OAuth authorization-code exchange (`oauthIntegrationLayer.cjs:264` `handleCallback`), refresh (`:325` `refreshToken`), and revoke (`:342`) are genuinely implemented over raw https for google/github/slack/notion/microsoft/linkedin.
- `.env` presence checked by variable-name grep only; no secret values read. `.env` holds real keys for: OPENAI, GROQ, RAZORPAY (5 vars), TELEGRAM (3), LINKEDIN (3), GOOGLE (1: `GOOGLE_API`). `.env` has NO Anthropic, Gemini, Slack, Discord, Stripe, Firebase, Supabase, Cloudflare, AWS, WhatsApp, PayPal keys.
- Live `/integrations/scan` reported **57 connectors**: CONNECTED 4, READY 43, PARTIAL 2, MISSING 8.

## Most surprising mismatch
No connector falsely reports "healthy" over a stub — the honesty of the framework is itself the finding. The nearest mismatch is **Google OAuth reporting `CONNECTED` in code when only a PUBLIC discovery endpoint (`accounts.google.com/.well-known/openid-configuration`) is probed** (`integrationConnectors.cjs:709-711`): that URL returns 200 for anyone on Earth regardless of whether `GOOGLE_CLIENT_ID/SECRET` are real, so a CONNECTED there proves only "Google is online," not "our OAuth app works." Same soft-verification for Microsoft (`:733`) and LinkedIn (`:750`). (In this .env it's moot — no Google creds set, so it shows READY.) The second surprise: **Anthropic and Gemini have fully-real, correct API code but their credential variable names are missing from `.env.example`**, so a by-the-book deployment would silently run without the two flagship non-OpenAI LLM providers.

## Fixes applied
None. No unwired-but-working code with a trivial bug was found; all real code paths are already wired. Remaining gaps require real credentials or new integration code (Gmail/Calendar/Drive API calls, PayPal from scratch), which is out of scope for execution-mode verification.
